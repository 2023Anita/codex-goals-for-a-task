# 并行目标用于一项任务

使用此技能将原始用户任务转化为可执行的计划简报、顶层目标草稿和并行子目标提示。默认不要执行任务、不要创建真实 Codex goal、不要派发代理；只有用户明确要求执行时才继续。

## 执行模式判断

先判断用户意图：

- 只要求生成 goal、并行目标、构建简报、代理提示或执行计划时，只输出计划，不执行任务。
- 明确要求“创建 goal”“开启 goal”“用 goal 执行”“按这个计划推进”时，才尝试创建顶层 goal 并执行。
- 只是要求审阅、改进或评估本 skill 时，保持只读分析，除非用户明确要求写文件。

## 流程概览

```text
用户任务 -> 填写简报 -> 生成目标草稿 -> 拆分子目标 -> 等待确认或执行 -> 综合 -> 最终回复
```

## 第一步：填写简报

将用户请求翻译到最合适的模板中。不要保留括号占位符。

可视化、前端、应用或内容构建任务使用：

```text
在[技术/框架]中构建[事物]。它应包括[主要功能]，并具有[交互/动画/行为细节]。使其具有[氛围/品质]，使用[视觉细节]、[环境细节]和[额外效果]。输出为[格式/文件类型]。
```

通用工程、调试、审计、研究或规划任务使用：

```text
在[项目/环境]中完成[核心任务]。范围包括[主要工作项]，需要满足[行为/质量标准]，遵守[约束/边界]。最终输出为[代码变更/分析报告/计划/验证结果]，并通过[验证方式]证明完成。
```

提问原则：

- 仅在缺失细节使任务无法执行或存在明显风险时向用户提问。
- 否则根据当前项目上下文保守推断，并标注关键推断。

## 第二步：生成顶层目标草稿

在派发或执行之前，定义任务完成标准。默认只生成目标草稿，不创建真实 Codex goal。

顶层目标必须包括：

- 已填写的简报。
- 具体、可验证的完成标准。
- 预期最终产出或答案。
- 汇报前应进行的验证。

目标格式：

```text
目标：[简明描述任务]

完成标准：
- [具体可衡量的标准1]
- [具体可衡量的标准2]

最终产出：[描述预期结果]

验证要求：
- [验证项1]
- [验证项2]
```

## 第三步：Goal 创建边界

Skill 触发不等于创建真实 goal。

- 默认行为：只生成计划简报、顶层目标草稿和子目标提示。
- 只有用户明确要求“创建 goal”“开启 goal”“用 goal 执行”时，才调用 Codex goal 工具。
- 如果没有可用 goal 工具，输出可复制的 goal 文本并继续用普通计划推进。
- 如果已有活动 goal 且无法创建新 goal，不要阻塞；将新目标写入当前工作计划。
- 子代理提示中的 `/goal ...` 只是提示格式，不保证平台会创建真实 goal。

## 第四步：并行分发

只拆分真正独立、并行执行有收益的工作。小任务由主代理直接完成。

合理并行度：

| 任务复杂度 | 建议代理数 | 示例场景 |
| --- | --- | --- |
| 简单 | 0-2 个 | 单一功能、单文件修改、只需计划 |
| 中等 | 3-5 个 | 多模块功能、设计加实现加验证 |
| 复杂 | 5-8 个 | 完整系统、多个子系统或多方向研究 |

不要并行的情况：

- 任务很小，拆分成本高于执行成本。
- 需要连续理解同一段代码或同一处状态。
- 涉及敏感凭证、生产写操作、删除或批量修改。
- 子目标高度耦合，结果难以安全合并。
- 用户只要求生成计划、简报或目标文本。

可拆分方向包括：

- 需求澄清和范围收敛。
- 架构、数据模型或集成规划。
- UI、交互或内容设计。
- 独立模块或文件实现。
- 测试、验证和边界情况审查。
- 文档、示例或最终说明。

子目标提示格式：

```text
/goal [一个明确的子目标]

上下文：
[完整且相关的约束条件。]

交付物：
[主代理需要返回的具体输出。]

边界：
[该代理拥有的文件、模块或决策，以及应避免的范围。]

验证：
[该代理应运行的检查，或应提供的推理和置信度。]
```

依赖处理：

- 如果子目标 A 依赖子目标 B 的输出，将它们串行化。
- 如果不确定是否有依赖，先派发确定独立的部分。
- 后续子目标应明确说明依赖来源和可用输入。

## 第五步：工具能力检查

在派发代理前，确认当前环境是否提供可用的多代理工具。

- 如果可用，并发派发真正独立的子目标。
- 如果不可用，输出可复制的子代理提示，或并行运行本地只读检查命令。
- 不要假设 `/goal` 文本会自动创建平台级 goal。
- 不要在等待代理时修改同一批文件，避免结果合并冲突。

## 第六步：综合

主代理拥有最终结果。当代理结果返回时：

1. 与项目上下文对比：直接读取相关文件或运行检查验证代理声明。
2. 解决冲突：优先级为测试结果 > 代码上下文 > 代理推断。
3. 选择性应用：只采用符合用户请求和项目约束的部分。
4. 控制范围：拒绝无关的顺便重构。
5. 验证结果：运行能够证明结果有效的最小可靠验证。

如果代理返回未经验证的声明，在依赖之前先验证。无法判断的内容标记为“需要用户确认”。

## 第七步：错误处理与恢复

子代理失败时：

- 临时问题可以最多重试一次。
- 结构性问题应重新设计子目标。
- 不可恢复时记录缺口，继续完成其他部分，并在最终回复中说明。

综合后发现结果不完整时，只派发针对性补充子目标，不全量重做。

## 第八步：最终回复

如果只是生成计划，最终回复应包括：

- 已填写的简报。
- 顶层目标草稿。
- 子目标列表或子代理提示。
- 建议的验证方式。
- 明确说明：尚未创建真实 Codex goal，等待用户确认。

如果用户明确要求执行，最终回复应包括：

- 已完成的结果。
- 产生或修改的文件。
- 运行过的验证。
- 未完成或需要用户确认的事项。

保持简洁，面向用户结果，不复述内部过程。
